# ТЗ: Полноценный конструктор диалогового тренажера (Scenario Editor)

Статус: Draft  
Версия: 1.0  
Дата: 2026-03-02

## 1. Цель
Реализовать полноценный конструктор сценариев после создания карточки тренажера в `/builder`.

Ключевое поведение:
- после создания карточки тренажера пользователь автоматически попадает в конструктор;
- конструктор представляет собой `flow/node-based` канвас с `drag&drop`, соединениями и редактором параметров выбранной ноды.

## 2. Контекст и область
Входит в реализацию:
- новый экран конструктора `GET /builder/dialog/:id` (замена текущего placeholder);
- канвас сценария (ноды + связи);
- инспектор справа (редактирование полей выбранной ноды);
- правила графа и валидация перед публикацией;
- сохранение черновика;
- preview/publish действия из конструктора.

Не входит в этот этап:
- совместное live-редактирование несколькими авторами в реальном времени;
- сложная медиа-обработка (транскодинг видео, waveform-редактор и т.п.);
- шаблоны сценариев и marketplace.

## 3. Пользовательские роли
- `Owner`, `Editor`: полный доступ к редактированию сценария.
- `Viewer`: только просмотр, без изменения.

## 4. Основные пользовательские сценарии
1. Пользователь в `/builder` нажимает `Create New Project`.
2. Система создает новый dialog/scenario в статусе `draft`.
3. UI автоматически делает redirect на `/builder/dialog/:id`.
4. Пользователь собирает сценарий на канвасе:
   - добавляет ноды;
   - соединяет связи;
   - настраивает параметры в правом инспекторе.
5. Пользователь сохраняет черновик.
6. Пользователь запускает `Preview`.
7. Пользователь публикует (`Publish`) только при прохождении валидации.

## 5. UX/дизайн требования (по референсу)
Базовая компоновка:
- верхний header: название сценария + действия `Preview`, `Publish`, `Share`;
- левый вертикальный toolbar: инструменты канваса;
- центральная область: канвас с сеткой, нодами и коннекторами;
- правый inspector: свойства выбранной ноды;
- нижний контрол масштаба.

Требования:
- светлая тема (без dark mode);
- интерактивные состояния: `hover/selected/disabled/error`;
- визуально понятные типы нод (цветовой код + иконки);
- на канвасе должны быть видны точки входа/выхода связей;
- поддержка drag&drop ноды по канвасу;
- зум/пан без “ломающихся” соединительных линий.

## 6. Обязательные типы нод

### 6.1 Start
Назначение:
- точка входа сценария.

Правила:
- строго одна нода `Start` на сценарий;
- ноду `Start` нельзя удалить;
- у `Start` нет входящих связей;
- минимум одна исходящая связь.

### 6.2 Message
Назначение:
- сообщение персонажа/системы (контекст).

Поля:
- `speakerType` (`npc` | `system`);
- `speakerName` (для `npc`);
- `text` (обязательное);
- `mediaRef` (опционально).

Правила:
- минимум 1 входящая связь (кроме случаев, когда сразу после Start);
- минимум 1 исходящая связь к `Response` или `End`.

### 6.3 Response (ключевой)
Назначение:
- вариант ответа ученика.

Поля:
- `label`/`responseText` (обязательное);
- `scoreDelta` (integer, может быть отрицательным);
- `feedbackType` (`positive` | `neutral` | `negative`);
- `hintEnabled` (boolean);
- `hintText` (обязательное, если `hintEnabled=true`);
- `nextStepRef` (обязательное).

Правила:
- `Response` должна иметь ровно 1 входящую связь (обычно из `Message`);
- минимум 1 `Response` на каждый branch, где ожидается выбор ученика;
- обязательно должна вести в следующую ноду (`Message` или `End`).

### 6.4 End
Назначение:
- завершение ветки сценария.

Поля:
- `endingType` (`success` | `fail` | `neutral`);
- `title` (обязательное);
- `description` (опционально).

Правила:
- у `End` нет исходящих связей;
- сценарий должен содержать минимум одну `End` ноду;
- из `Start` должна существовать достижимая `End`.

## 7. Функциональные требования

### 7.1 Навигация и создание
- FR-001: после `POST /api/v1/builder/dialogs` клиент автоматически переходит на `/builder/dialog/:id`.
- FR-002: если `id` не существует или нет доступа, показывать error-state.

### 7.2 Канвас
- FR-003: перемещение нод drag&drop.
- FR-004: соединение нод через коннекторы (drag handle -> target handle).
- FR-005: удаление связи.
- FR-006: zoom in/out (минимум 25%, максимум 200%) и pan.
- FR-007: выделение ноды (single-select), подсветка выбранной.
- FR-008: автопозиционирование новых нод в видимой области канваса.

### 7.3 CRUD нод
- FR-009: добавление ноды через toolbar или контекстное действие.
- FR-010: редактирование полей выбранной ноды в правом inspector.
- FR-011: удаление ноды с проверкой ограничений (`Start` нельзя удалять).
- FR-012: при удалении ноды все связанные edge удаляются каскадно (кроме запрещенных действий).

### 7.4 Сохранение
- FR-013: кнопка `Save Changes` сохраняет изменения в draft.
- FR-014: dirty-state (индикатор несохраненных изменений).
- FR-015: защита от потери данных при уходе со страницы (browser prompt, если dirty).

### 7.5 Preview/Publish
- FR-016: `Preview` запускает draft-модель в runtime preview-mode.
- FR-017: `Publish` вызывает валидацию графа и контента.
- FR-018: если есть blocking-ошибки, публикация запрещена, ошибки показываются с привязкой к нодам.

## 8. Правила графа и валидации

Блокирующие ошибки:
- V-001: отсутствует `Start`;
- V-002: больше одной `Start`;
- V-003: `Start` без исходящей связи;
- V-004: отсутствует `End`;
- V-005: недостижимые ноды (не достижимы из `Start`);
- V-006: нода без обязательных полей (`text`, `responseText`, `nextStepRef`, и т.д.);
- V-007: `End` с исходящей связью;
- V-008: `Response` без входящей или исходящей связи;
- V-009: ссылка `nextStepRef` указывает на несуществующую ноду.

Предупреждения:
- W-001: очень длинный текст (`Message.text > 1000`);
- W-002: слишком много исходящих ответов из одного сообщения (`>10`);
- W-003: есть циклы в графе (допустимо, но показывать предупреждение).

## 10. API контракт (этап конструктора)

### 10.1 Получить модель конструктора
`GET /api/v1/builder/dialogs/:id/editor`

Response:
- `200` + JSON модели (`nodes`, `edges`, `meta`).

### 10.2 Сохранить модель конструктора
`PUT /api/v1/builder/dialogs/:id/editor`

Request:
- полный snapshot `nodes + edges + meta`;
- `revision` для optimistic locking.

Response:
- `200` + новая `revision`.

### 10.3 Валидация перед публикацией
`POST /api/v1/builder/dialogs/:id/editor/validate`

Response:
```json
{
  "valid": false,
  "errors": [
    { "code": "V-006", "nodeId": "node_resp_a", "message": "responseText is required" }
  ],
  "warnings": []
}
```

### 10.4 Схема текущего backend
Текущее API `publish/unpublish/export` сохраняется. Перед `publish` дополнительно выполняется validate editor-модели.

## 11. Изменения БД (Supabase)

Минимально необходимо:
- использовать существующие таблицы `scenario_steps`, `scenario_choices`, `scenario_endings`;
- добавить хранение позиций нод:
  - вариант A: новое поле `layout_json` в `scenario_versions`;
  - вариант B: отдельная таблица `scenario_layout_nodes`.

Рекомендация MVP:
- добавить `layout_json` в `scenario_versions` (быстрее внедрение).

Содержимое `layout_json`:
- `nodes[]` (id/type/x/y);
- `edges[]` (id/source/target);
- служебные UI-параметры (zoom, viewport).

## 12. Требования к производительности
- P-001: открытие конструктора до first interactive <= 2.0s на сценарии до 100 нод.
- P-002: операции drag/connect <= 16ms на кадр (целевой 60fps) до 200 нод.
- P-003: сохранение draft <= 1.0s в нормальной сети.

## 13. Требования по безопасности и доступу
- доступ по `workspace_members` и RLS;
- только `owner/editor` могут изменять editor-модель;
- sanitize текстовых полей (защита от XSS в preview/runtime);
- лимиты размеров полей и payload.

## 14. Логирование и аналитика
События для telemetry:
- `editor_opened`;
- `node_added`;
- `node_deleted`;
- `edge_connected`;
- `editor_saved`;
- `editor_validate_failed`;
- `editor_published`.

Назначение:
- анализ UX-узких мест и частых ошибок валидации.

## 15. Acceptance Criteria

### AC-1 Создание и автопереход
- при создании карточки в `/builder` пользователь попадает на `/builder/dialog/:id` автоматически;
- если создание неуспешно, переход не выполняется и показывается ошибка.

### AC-2 Базовый редактор графа
- можно добавить/переместить/удалить `Message`, `Response`, `End`;
- `Start` присутствует всегда, удалить его нельзя;
- можно создавать и удалять связи между нодами.

### AC-3 Инспектор ноды
- при выборе ноды справа отображаются ее поля;
- изменение полей сохраняется в draft после `Save Changes`;
- для `Response` доступны: текст, score, feedbackType, hint toggle/text, next step.

### AC-4 Валидация
- публикация блокируется при нарушении правил V-001..V-009;
- ошибки показываются с привязкой к конкретной ноде;
- после исправления ошибок публикация доступна.

### AC-5 Стабильность данных
- после перезагрузки страницы сохраняются структура графа, позиции нод, связи и поля;
- нет потери данных при корректном сохранении.

### AC-6 Доступы
- `viewer` не может редактировать или публиковать;
- `editor/owner` имеют полный доступ в пределах своего workspace.

## 16. Этапы реализации

### Этап A — каркас editor страницы
- route `/builder/dialog/:id`;
- layout (header/toolbar/canvas/inspector);
- загрузка модели с backend.

### Этап B — граф и взаимодействия
- node/edge rendering;
- drag&drop;
- select/delete/connect;
- zoom/pan.

### Этап C — inspector и сохранение
- формы полей по типам нод;
- `Save Changes`;
- dirty-state + guard.

### Этап D — валидация и publish
- backend validate endpoint;
- UI ошибок;
- интеграция с publish flow.

### Этап E — QA и стабилизация
- тест-кейсы на правила графа;
- проверка производительности;
- фиксация edge cases.

## 17. Риски и меры снижения
- Риск: сложность UX при больших графах.
  - Мера: миникарта, fit-to-screen, auto-layout (в следующем этапе).
- Риск: конфликт структуры `steps/choices/endings` и canvas-модели.
  - Мера: единый adapter слой преобразования `DB <-> editor graph`.
- Риск: ошибки соединений при массовом редактировании.
  - Мера: централизованная валидация графа на frontend и backend.
